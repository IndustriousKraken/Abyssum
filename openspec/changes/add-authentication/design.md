# Design: Authentication

## Technical Approach

Authentication lives in `abyssum-core` (an `auth` module) so both the web surface and any
future surface call one implementation. The web layer (`add-web-interface`) consumes it via
middleware; this change specifies only the engine-level behavior.

Three persisted concepts, all in the SQLite store from `add-result-persistence`:

```
users     (id, username UNIQUE, password_hash, role, created_at)
sessions  (token, user_id, created_at, expires_at, last_seen_at)
```

The existing scan-session record (owned by `add-result-persistence`) gains an
`owner_user_id` column populated at creation. This change defines the **meaning** of that
column (ownership + visibility); persistence merely stores it.

```
register(username, password):
    if first user -> role = admin else role = user
    hash = password_hash(password, fresh_salt)
    insert user(username, hash, role)        # UNIQUE rejects duplicates

login(username, password):
    user = lookup(username)
    if !user or !verify(password, user.hash): reject (constant message)
    token = random_opaque_token()
    insert session(token, user.id, expires_at = now + lifetime)
    return token

authorize(token):
    s = lookup(token)
    if !s or s.expired: deny
    refresh last_seen_at (idle-timeout)
    return s.user
```

## Library / Crate Choices

- **Password hashing:** **`argon2`** (the `argon2` crate, Argon2id) via the
  `password-hash` traits. Argon2id is the current OWASP-recommended default; memory-hard and
  resistant to GPU cracking. `bcrypt` was considered but rejected — Argon2id is the stronger
  modern default and the crate handles salt generation and the encoded PHC string (salt +
  params embedded) for us, so we never store a bare hash or manage salts by hand.
- **Salt:** per-password random salt generated by the hashing crate (via `OsRng`), embedded
  in the stored PHC string. No global pepper in v1 (can be added later as a config-sourced
  secret without changing observable behavior).
- **Session tokens:** opaque, high-entropy random tokens from a CSPRNG (`rand`'s `OsRng`).
  Tokens are the lookup key; only their value (never a password) travels with requests.
- **Session storage:** **server-side in SQLite** (the `sessions` table above), not stateless
  JWTs. Rationale: server-side sessions can be revoked immediately (logout, admin action,
  expiry sweep) and we already have SQLite; JWTs would add a signing-secret and a revocation
  problem for no v1 benefit. No Redis (canon non-goal).
- **Constant-time verify:** rely on the hashing crate's `verify_password`, which compares in
  constant time; the login error message is identical for unknown-user and wrong-password.

## Architecture Decisions

### Decision: Roles are a closed set `{admin, user}`
Just enough for "owner-only + admin-sees-all". No per-resource ACLs, no teams/sharing/SSO
(canon defers those). The role is a column on the user, set once at registration.

### Decision: First registered user becomes admin
Canon open-question default (`project.md` #3). Simplest zero-config bootstrap: whoever stands
the instance up and registers first is admin; everyone after is a regular user. Flagged as an
assumption. (Alternative — env-seeded admin — is a later `MODIFIED` if the human prefers it.)

### Decision: Ownership is immutable and set at creation
The creating user's id is stamped onto the scan session when it is created and never changes.
Visibility is a read-time filter, not a copy of data, so an admin always sees the live set.

### Decision: Sessions expire by absolute lifetime + idle timeout
Both an absolute max age and an idle timeout (refreshed on use) bound a session. Expired
sessions are treated as absent; a periodic/lazy sweep removes them. Lifetimes are
config-sourced (this change reads them; key names are config detail, not behavior).

## Testing

- Unit: registering stores a hash that is **not** the plaintext and that `verify` accepts;
  two registrations of the same password produce **different** stored hashes (random salt).
- Unit: first registrant gets `admin`, second gets `user`; duplicate username rejected.
- Unit: login with wrong password and login with unknown user both fail with the **same**
  error; correct credentials yield a session token.
- Unit: an expired session is rejected; logout invalidates a session immediately.
- Unit/integration: a regular user sees only sessions they own; an admin sees all; a request
  with no/invalid session to protected functionality is denied. All in-process with an
  in-memory/temp SQLite DB. **No network, no real targets.**
